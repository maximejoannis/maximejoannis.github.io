Desktop Automation Electron App Reverse Engineering

AMN Auditor — Ingénierie d’automatisation

Objectif : rendre testables et automatisables des parcours critiques sur une application desktop Electron, en combinant UI Automation Windows et extraction DOM via DevTools.

Contrainte principale : sur une app Electron, l’arbre UIA Windows remonte souvent un “Chrome Legacy Window” avec peu d’éléments exploitables. → solution : hybride (UIA / WinAppDriver pour le shell + DevTools pour le DOM React).

Architecture de l’automatisation

Vue système

L’objectif est de garder une traçabilité entre : l’extraction du DOM, l’inventaire des éléments, puis l’écriture de scripts (Playwright / WinAppDriver). 

AMN Auditor (Electron Desktop)
    ├─ Windows layer (UIA / WinAppDriver)
    │    ├─ inventory UIA (pywinauto)
    │    └─ attach WinAppDriver via window handle
    └─ Web layer (DOM React via DevTools)
         ├─ remote debugging port (9222)
         ├─ DOM extraction (CDP / WebSocket)
         └─ selectors extraction (IDs / interactives)
Exemple concret : deux boutons clés identifiés via extraction DOM : #home-create-project-btn et #home-open-project-btn. :contentReference[oaicite:2]{index=2}

Workflow (du “black box” au test auto)

1. Lancer l’app
Vérifier que l’UI est stable (loader terminé).
2. Exposer DevTools
Remote debugging (port 9222), puis connexion CDP/WebSocket.
3. Extraire le DOM + interactifs
Générer rapports : hiérarchie + interactifs filtrés.
4. Écrire les scripts
Playwright Electron pour le DOM ; WinAppDriver/UIA pour la fenêtre.
5. Stabiliser
Attentes, sélecteurs robustes, réduction flakiness.
RÈGLE D’OR

Si UIA ne remonte pas le DOM React, ne pas forcer : passer sur DevTools/Playwright.

Stratégie de sélection

Priorité Sélecteur Pourquoi
P1 #id Stable, unique, rapide
P2 [name="..."] Souvent stable côté formulaires
P3 text= / role= Lisible mais fragile (i18n / UI)
P4 .class... Souvent dynamique (framework)
CONSEIL

Pour limiter la flakiness : privilégier ID / Name, et encapsuler les attentes (loader, état detached/visible).

Checklist de stabilisation

  • Attendre la disparition du loader (état detached) avant d’interagir
  • Préférer les sélecteurs #id aux classes UI dynamiques
  • Centraliser les timeouts et les “waits” dans des helpers
  • Tracer les étapes clés (logs) + capturer screenshot sur échec
  • Isoler les flows : smoke (N0) → non-régression (N1)
