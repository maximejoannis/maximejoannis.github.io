L'approche de test choisie pour les tests système (réalisés par les testeurs professionnels)

L'approche de test choisie est : Les Tests basés sur les risques (Risk Based Testing - RBT)

Cette approche consiste à identifier les risques potentiels et les exigences du projet, puis à élaborer des plans de test qui ciblent spécifiquement ces aspects critiques. En concentrant les efforts de test sur les domaines présentant les plus grands risques et les exigences les plus importantes, RBT maximise l'efficacité des activités de test et garantit une couverture adéquate des fonctionnalités essentielles du projet.

Plus le risque de défaillance sur une exigence est important plus grand sera l'effort de test alloué à cette exigence. Pas de risque ==> Pas de test

Exigences P1 :
Tests manuels : Couvrir à la fois les cas nominaux et les cas d'exception.
Les cas de test manuels sont conçus et implémentés.
Tests de non régression (automatisés) : Inclure à la fois les cas nominaux et les cas d'exception.

Objectifs de qualité :
- Atteindre 100% de réussite pour tous les cas de test.
- Aucun défaut bloquant ne doit être présent.
- Aucun défaut majeur ne doit être présent.
- Limiter les défauts mineurs à moins de 6.
Ces tests doivent être prioritaires et exécutés en priorité pour détecter les défauts critiques.

Exigences P2 :
Tests manuels : Couvrir à la fois les cas nominaux et les cas d'exception.
Les cas de test manuels sont conçus et implémentés.
Tests de non régression (automatisés) : Se concentrer sur les cas nominaux.

Objectifs de qualité :
- Atteindre 100% de réussite pour tous les cas de test.
- Ne pas dépasser 1 défaut majeur.
- Limiter les défauts mineurs à moins de 12.
Ces tests doivent être effectués après ceux des P1, en mettant l'accent sur les cas essentiels.

Exigences P3 :
Tests manuels : Se concentrer sur les cas nominaux.
Les cas de test manuels sont conçus sans être implémentés.
Tests de non régression (automatisés) : Non applicables.

Objectifs de qualité :
- Viser à ce que 50% des cas de test soient réussis.
- Aucun défaut bloquant n'est acceptable.
- Les défauts majeurs sont tolérés mais doivent être documentés.
- Les défauts mineurs sont tolérés mais doivent être surveillés.
Ces tests sont moins prioritaires, mais doivent tout de même être réalisés pour couvrir une partie des fonctionnalités moins critiques.

Cette stratégie permet de prioriser les tests en fonction de l'importance des fonctionnalités et de garantir une détection précoce des défauts majeurs tout en optimisant les ressources de test.

Détail des exigences

# Résumé Description Critères d'acceptation
7 En tant que Voyageur Je souhaite envoyer une demande de réservation Afin de bénéficier d’un séjour dans un logement saisonnier
8 En tant que Hôte Je souhaite accepter ou refuser une demande de réservation Afin de planifier les séjours dans mon logement

Un utilisateur connecté en tant que Voyageur, peut suite à la consultation d’une annonce qui répond à ses critères vouloir réserver le logement proposé dans cette annonce.

L'annonce doit être créée auparavant par un utilisateur de type Hôte.

Un utilisateur non connecté peut entamer la procédure de la demande de réservation mais le système ne doit pas la laisser aboutir.

Pour réaliser une demande de réservation, le Voyageur doit renseigner les informations suivantes :

  • Date de début de réservation : le système ne doit proposer que les dates disponibles dans le futur
  • Date de fin de réservation : le système ne doit proposer que les dates disponibles dans le futur, postérieure à la date de début
  • Nombre de voyageurs : entier non nul
  • Message adressé à l’hôte

Une période réservée est une période bloquée à d’autres réservations.

Quand le Voyageur a envoyé sa demande de réservation, l’hôte correspondant à l’annonce doit recevoir :

  • 1 nouveau message dans sa boite de réception « Messages »
  • 1 nouvelle réservation dans sa boite « Réservations »

Un hôte reçoit les demandes de réservations émanant des voyageurs. Chaque nouvelle demande reçue doit avoir le statut : NOUVEAU.

Les informations suivantes doivent être affichées :

  • Date de début
  • Date de fin
  • Voyageurs
  • Animaux domestiques
  • Total

Le système doit donner à l’hôte la possibilité de « Confirmer » la réservation si ce dernier voudrait donner suite à la demande.

Cliquer sur le bouton « Confirmer » doit transférer l’hôte vers une page avec les détails de paiement :

  • Nombre de nuits
  • Prix par nuit
  • Taux d’imposition

Le système doit donner les possibilités d’actions suivantes à l’hôte :

  • Confirmer la disponibilité : Le nouveau statut de la demande sera PAIEMENT EN ATTENTE dans le tableau de bord de l’hôte et DISPONIBLE dans le tableau de bord du Voyageur
  • Refuser : permet de refuser une demande de réservation avec la possibilité de justifier la décision. Le nouveau statut de la demande sera REFUSÉ.
  • Ajouter des frais supplémentaires : ajouter un ou plusieurs frais supplémentaires avec nom et valeur.
  • Faire une remise : ajouter un ou plusieurs remises avec nom et valeur.

Le système permet au voyageur de Payer pour avancer sur une demande de réservation avec un statut DISPONIBLE via un bouton « Payer maintenant ».

En choisissant de payer, une page de paiement détaillée est affichée avec les informations suivantes :

  • Nombre de nuits
  • Prix par nuit
  • Taux d’imposition

Le système doit donner les possibilités d’actions suivantes au voyageur :

  • Payer : Un seul mode de paiement « Hors site »
  • Annuler : Le nouveau statut de la demande sera ANNULÉ.

Si les informations de paiement de l’hôte ne sont pas initialisées, le système ne permet pas le paiement.

Pour initialiser les informations de paiement de l’hôte, il faut aller dans le menu Profil > Paiement et renseigner les informations nécessaires.
